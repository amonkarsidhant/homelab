# ADR-001 — Local-First Agent Runtime

**Status:** Accepted  
**Date:** 2026-03-27  
**Author:** CTO (jarvis-cto)  
**Supersedes:** None  

---

## Context

JARVIS Home OS is an ambient home intelligence platform. The agent must reason about the home, learn family patterns, and act within guardrails — all while satisfying three hard constraints: **privacy-first** (cloud is opt-in only), **offline-capable** (home works without internet), and **self-hosted** (runs on homeowner hardware).

This ADR establishes where the agent brain runs, how it persists memory, and how guardrails are enforced.

---

## Decision

**JARVIS uses a tiered local-first inference architecture:**

| Tier | Where It Runs | What It Does |
|------|--------------|--------------|
| **Edge (home hub)** | Raspberry Pi 4/5 — Rust runtime | Wake word, STT, intent classification, device control, pattern storage |
| **On-device model** | Llama.cpp or whisper.cpp (quantized) | NL understanding, command generation, guardrail evaluation |
| **Cloud (opt-in)** | JARVIS Cloud — only if user enables Home+ | Cross-device learning, remote assist, firmware updates |

**No mandatory cloud dependency exists for any core function.**

---

## Why Local-First

### Privacy
- Voice commands never leave the home network unless the user explicitly opts into cloud assist
- Family patterns (routines, preferences) are stored in local SQLite on the hub
- Even Home+ cloud sync uses end-to-end encryption with user-controlled keys

### Latency
- Round-trip voice → action on local hub: target <300ms
- Same round-trip via cloud: 800ms–2000ms (network dependent)
- Offline mode must have identical latency to online mode

### Offline
- 100% of P0 features (voice, Matter control, pattern learning) work without internet
- The hub discovers and controls devices over LAN, not via cloud relay
- Home+ cloud sync reconciles state on reconnect

## Inference Stack

### Speech-to-Text
- **Primary:** whisper.cpp (tiny.en or base.en model) — runs on Pi CPU
- **Rationale:** OpenAI Whisper is proven, quantized variants run at 8-15x realtime on Pi 4
- **Model size:** ~75MB for tiny-en, ~150MB for base-en
- **Latency target:** <200ms for command-length audio

### Intent Classification / NL Understanding
- **Primary:** Llama.cpp with a fine-tuned instruction-following model (3B–7B parameter, 4-bit quantized)
- **Running on:** Pi GPU (if available) or CPU with SIMD acceleration
- **Rationale:** Ollama has established patterns for local LLM inference on edge hardware
- **Fallback:** Rule-based intent parser (deterministic, no model) for when inference is too slow

### Agent Runtime
- **Rust core:** Device communication, scheduling, state management — compiled, memory-safe, hard real-time
- **Python agent logic:** Intent handlers, pattern learning, guardrail evaluation — interpreted, flexible, iterates fast
- **IPC:** Unix domain sockets or gRPC between Rust daemon and Python agent process

### Model Storage
- Models stored on microSD or attached USB storage (not network storage)
- Versioned: each model ships with a manifest (hash, version, min hardware spec)

## Memory Persistence

### Household State (SQLite)
- Device registry (Matter node ID, capabilities, state)
- Room topology (rooms, devices per room)
- User accounts and roles (parent, child, guest)
- Session logs (voice commands, executed actions, timestamps)

### Learned Patterns (SQLite)
- Time-based routines (learned from repeated manual actions)
- Presence patterns (motion → absence events)
- Device affinity (which devices are typically used together)

### Schema

```sql
CREATE TABLE devices (
  matter_id TEXT PRIMARY KEY,
  name TEXT,
  room_id TEXT,
  type TEXT,
  state_json TEXT,
  last_seen INTEGER
);

CREATE TABLE patterns (
  id TEXT PRIMARY KEY,
  type TEXT, -- 'time', 'motion', 'sequence'
  trigger_json TEXT,
  action_json TEXT,
  confidence REAL,
  created_at INTEGER,
  enabled INTEGER
);

CREATE TABLE household_members (
  id TEXT PRIMARY KEY,
  name TEXT,
  role TEXT, -- 'parent', 'child', 'guest'
  pin_hash TEXT,
  created_at INTEGER
);
```

### Pattern Learning Algorithm
- v0.1: Simple frequency counting — "you've done X at Y time Z times → suggest as routine"
- Future (Phase 2): Temporal convolutional networks, bayesian structure learning
- User must approve or delete all learned patterns — no silent automation

## Guardrail Enforcement

Guardrails are enforced at three layers:

### 1. Policy Layer (Rust — enforced before action)
```
class GuardrailPolicy:
  - children cannot: control locks, modify parental settings, disable cameras
  - guests cannot: add/remove devices, modify household settings, view history
  - all users: no single action >$50 estimated impact (prevents accidental large orders)
  - agent: never initiates outbound network calls without explicit user command
```

### 2. Intent Validation (Python — evaluated per command)
- Incoming intent JSON is validated against allowed action set
- Action parameters are schema-checked (e.g., brightness 0–100, not -5 or 300)
- If guardrail fires → intent is blocked, user is notified via voice + app notification

### 3. Audit Log (SQLite — immutable append-only)
```
CREATE TABLE audit_log (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp INTEGER,
  user_id TEXT,
  intent_json TEXT,
  guardrail_result TEXT, -- 'allowed', 'blocked', 'modified'
  blocked_reason TEXT,
  execution_time_ms INTEGER
);
```
Audit log is local only. Home+ cloud does not replicate it unless user explicitly enables.

## Consequences

### Positive
- All core functions work offline, satisfying CEO-001 first principles
- Privacy is architecturally enforced, not policy-enforced
- Rust core ensures deterministic, low-latency device control
- Python agent logic enables fast iteration on ML features

### Negative
- Pi 4 may struggle with 7B model + Whisper simultaneously (mitigation: Pi 5 or use 3B model)
- Local model quality is lower than GPT-4 class cloud models
- Pattern learning v0.1 is primitive (frequency counting only)

### Risks
- **Mitigation:** Hardware tiering document (STACK.md) will specify Pi 4 minimum, Pi 5 recommended
- **Risk:** Model updates require hub reboot — mitigated by A/B model loading

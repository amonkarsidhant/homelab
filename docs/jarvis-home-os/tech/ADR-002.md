# ADR-002 — Matter Bridge Strategy

**Status:** Accepted  
**Date:** 2026-03-27  
**Author:** CTO (jarvis-cto)  
**Supersedes:** None  

---

## Context

JARVIS Home OS must connect to smart home devices. The Matter standard provides a unified IP-based protocol for device certification, but the real-world home still has Zigbee bulbs, Z-Wave locks, and legacy WiFi devices. This ADR defines how JARVIS discovers, pairs, and controls devices — both Matter and non-Matter.

---

## Decision

**JARVIS acts as a Matter facilitator (bridge + controller) rather than a full Matter hub.**

| Role | Description | Responsibility |
|------|-------------|----------------|
| **Matter Controller** | Manages Matter fabric, commissions devices | Pairing, fabric topology, device ownership |
| **Matter Bridge** | Exposes non-Matter devices to Matter fabric | Translates Zigbee/Z-Wave to Matter |
| **Thread Border Router** | Routes Thread packets over WiFi/Ethernet | Required for Thread devices |

**JARVIS ships with Thread Border Router capability on Pi 4+ (WiFi STA + Thread radio via 802.15.4 dongle or integrated).**

**Non-Matter devices are bridged via Home Assistant OS integration, not re-implemented.**

---

## Matter Device Support

### Discovery
- mDNS/Bonjour discovery on LAN for Matter devices advertising `_matter._tcp`
- Matter commissioning protocol for pairing new devices
- JARVIS maintains a local device registry (Matter node ID → device capability + state)

### Pairing Flow
```
1. User says "pair new device" or taps "Add Device" in app
2. JARVIS enters commissioning mode (BLE advertisement + OTA over WiFi)
3. Device QR code scanned via app → pairing token exchanged
4. Device joins JARVIS Matter fabric
5. Device metadata stored in local SQLite (name, room, type, capabilities)
```

### Control Protocol
- Matter interactions use Matter cluster library (on/off, level, color, temperature)
- Commands issued over LAN (IPv6 UDP) — no cloud relay
- Device state cached locally for offline queries ("are the living room lights on?")

### Supported Matter Clusters (v0.1)
| Cluster | Support Level |
|---------|---------------|
| OnOff | Full ( bulbs, plugs, switches) |
| LevelControl | Full (dimmable devices) |
| ColorControl | Full (color bulbs) |
| TemperatureMeasurement | Full (thermostats, sensors) |
| OccupancySensing | Full (motion sensors) |
| DoorLock | Basic (no biometric overrides in v0.1) |
| WindowCovering | Basic (open/close/position) |

## Non-Matter Device Strategy

### Why Home Assistant OS (HAOS)
- Re-implementing Zigbee, Z-Wave, and proprietary WiFi stacks would take years
- Home Assistant has battle-tested integrations for 2,000+ device brands
- HAOS runs as a container on the same Pi 4/5 that runs JARVIS
- HA's Matter integration already exposes non-Matter devices as Matter devices

### Architecture
```
┌─────────────────────────────────────────────────────┐
│  JARVIS Runtime (Rust + Python)                    │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────┐  │
│  │ Voice Agent │  │ Matter Stack │  │ App API    │  │
│  └─────────────┘  └──────┬───────┘  └────────────┘  │
│                          │                           │
│              ┌───────────┴───────────┐               │
│              │  Home Assistant OS    │               │
│              │  (Docker container)   │               │
│              │  ┌─────────────────┐ │               │
│              │  │ Zigbee2MQTT     │ │               │
│              │  │ Z-Wave JS UI    │ │               │
│              │  │ Tuya, Shelly    │ │               │
│              │  └─────────────────┘ │               │
│              └───────────┬───────────┘               │
└──────────────────────────┼───────────────────────────┘
                          │
              Matter Thread/WiFi/Ethernet
                          │
              ┌───────────┴───────────┐
              │  Matter Devices       │
              │  (IKEA, Nanoleaf,     │
              │   TP-Link, etc.)       │

```

### Communication Between JARVIS and HA
- HA exposes a WebSocket API (`ws://localhost:8123/api/websocket`)
- JARVIS subscribes to device state changes via HA websocket events
- JARVIS issues commands via HA REST API (`/api/services/switch/turn_on`)
- Both run on same host — no network latency

### Non-Matter Devices via HA
| Protocol | Integration | Exposed to JARVIS as |
|----------|-------------|---------------------|
| Zigbee | Zigbee2MQTT | Matter bulb or plug |
| Z-Wave | Z-Wave JS UI | Matter lock or sensor |
| WiFi (Tuya) | LocalTuya | Matter plug |
| WiFi (Shelly) | Shelly integration | Matter plug |
| Infrared | Broadlink RM | Matter HVAC or media |

## Local Network Communication Model

### No Inbound Ports
- JARVIS does NOT expose any inbound ports to the internet
- All remote access (when enabled) goes through JARVIS Cloud relay (WebRTC or TURN)
- This prevents NAT traversal issues and eliminates port forwarding requirements

### LAN Communication
- Matter: IPv6 UDP (port 5540) — device-to-hub
- HA API: localhost only (127.0.0.1) — hub-to-HA
- App-to-hub: TLS-over-HTTPS on port 8443 (self-signed cert generated at setup)

### Hub Discovery
- Mobile app discovers hub via mDNS (`_jarvis._tcp.local`)
- App connects to hub's local API via HTTPS (self-signed cert, cert fingerprint shown in app during pairing)
- This eliminates the need for cloud discovery or manual IP entry

## Thread Border Router

### What It Means
Thread is a low-power mesh networking protocol built on 802.15.4. Many Matter devices (especially sensors, locks, and thermostats) are Thread devices. They cannot communicate over WiFi directly — they need a Thread Border Router to bridge Thread packets to the home network.

### JARVIS as Thread Border Router
- **Hardware:** Raspberry Pi 4/5 + 802.15.4 USB dongle (Nordic nRF52840 or Silabs EFR32)
- **Software:** OpenThread daemon (ot-daemon) running as a system service
- **Certification:** JARVIS will pursue Thread Border Router certification via CSA (Connectivity Standards Alliance)

### Why Not Native Thread (Pi 5 integrated)?
- Pi 5 has Bluetooth 5.4 but no 802.15.4 integrated
- External dongle is the correct approach for v0.1
- Pi 5 with future 802.15.4 module could be "Thread Built-In" SKU

## Consequences

### Positive
- Home Assistant integration gives instant support for 2,000+ device brands
- Matter native + HA bridge covers 99% of home device scenarios
- No mandatory cloud — all device traffic stays on LAN
- Hub discovery via mDNS eliminates complex onboarding

### Negative
- Home Assistant OS adds ~2GB storage and 512MB RAM overhead
- Two update mechanisms (JARVIS + HA) instead of one
- HA breaking changes can affect JARVIS device control

### Risks
- **Mitigation:** HA runs as a pinned Docker container with a known-good image hash — updates are tested before rollout
- **Risk:** HA's Matter integration maturity — mitigated by Matter 1.2+ being production-stable

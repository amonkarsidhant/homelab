# ADR-003 — Mobile App Stack

**Status:** Accepted  
**Date:** 2026-03-27  
**Author:** CTO (nestra-cto)  
**Supersedes:** None  

---

## Context

Nestra Home OS requires a mobile companion app (iOS + Android) for device dashboard, room organization, manual control, and push notifications. The app must communicate securely with the home hub over LAN (when on same network) and remotely (via Nestra Cloud relay when enabled).

This ADR chooses the mobile framework, defines the hub communication protocol, and specifies offline sync strategy.

---

## Decision

**Recommendation: Flutter**

| Factor | Flutter | React Native | Native (Swift/Kotlin) |
|--------|---------|--------------|----------------------|
| Time to first feature | Fast — single codebase | Fast — single codebase | Slow — two codebases |
| Performance | Near-native (Skia) | Medium (JS bridge) | Native |
| Home automation UI maturity | Good (Material 3) | Medium | High (platform specific) |
| Nestra brand white-label | Excellent — theming engine | Medium — CSS-in-JS | Difficult — per-platform |
| Developer availability | Good (Dart, growing) | Excellent (JS/TS) | Medium |
| Smart home libraries | libpyrite, matter.js work | matter.js excellent | Matter SDK native |
| Offline-first architecture | SQLite + Provider viable | AsyncStorage + Context | Core Data + MVVM |

**Secondary choice: React Native** if the team has strong JS/TS expertise and Matter.js availability is the deciding factor.

---

## Why Flutter

### Brand White-Label is a Core Requirement
- The PRD specifies that decor brands (IKEA, Wayfair, Ashley) will white-label Nestra
- Flutter's theming engine supports runtime brand injection (colors, typography, icons, logo) with zero code changes
- React Native requires more per-platform tweaking to achieve equivalent brand consistency
- A single `BrandTheme` class can swap an entire app's visual identity at runtime

### Matter.js Ecosystem
- matter.js (JavaScript Matter protocol library) works with React Native via a WebView bridge
- Flutter has no first-class Matter SDK — a custom Flutter plugin wrapping libpyrite (Rust Matter library) is required
- **This is the strongest argument for React Native.** Mitigation: the Nestra app communicates with the hub, not directly with Matter devices. The hub handles Matter protocol. The app only calls REST/WebSocket APIs on the hub.

### UI Performance for Home Automation
- Home automation UIs are device-centric, not content-centric — lists of devices, grids of rooms, toggles
- Flutter's Skia renderer handles these patterns well
- React Native's JS bridge can cause jank on Android when updating large device lists rapidly

### Cross-Platform Consistency
- Families use iOS and Android equally
- Flutter renders identically on both — no "Android looks different" issues
- Native Swift/Kotlin would require two separate UI codebases and two QA passes

## Hub Communication Protocol

### Primary: Local LAN (Same Network)
```
┌─────────────┐         mDNS          ┌─────────────┐
│ Nestra App  │ ──── (_nestra._tcp) ──│ Nestra Hub  │
│  (Mobile)   │ ◄─── HTTPS (8443) ──► │  (Pi 4/5)   │
└─────────────┘      Self-signed       └─────────────┘
```
- App discovers hub via mDNS service discovery
- TLS connection to hub's local API (port 8443)
- Certificate fingerprint shown in app during pairing (QR code)
- All traffic stays on LAN — no internet required

### Secondary: Remote Access (Nestra Cloud Relay)
```
┌─────────────┐              ┌─────────────┐              ┌─────────────┐
│ Nestra App  │ ───────────►│ Nestra Cloud│ ◄───────────│ Nestra Hub  │
│  (Mobile)   │   TLS/WSS    │  (Relay)    │   TLS/WSS    │  (Pi 4/5)   │
└─────────────┘              └─────────────┘              └─────────────┘
```
- When user enables "remote access" (Home+ feature), app connects via Nestra Cloud relay
- WebRTC DataChannel or WebSocket relay — cloud never stores or inspects traffic
- Nestra Cloud relay is a passthrough, not a proxy — end-to-end encryption maintained
- Cloud relay is opt-in, never automatic

### API Design
```
GET  /api/v1/hub/status          — Hub info, firmware version, online devices
GET  /api/v1/devices             — List all paired devices
GET  /api/v1/devices/{id}        — Single device state
POST /api/v1/devices/{id}/control — Issue command { "command": "on" }
GET  /api/v1/rooms               — Room layout
GET  /api/v1/patterns            — Learned patterns
POST /api/v1/patterns/{id}/toggle — Enable/disable pattern
WS   /api/v1/events              — Real-time device state updates
POST /api/v1/auth/verify          — Verify household member PIN
```

### Push Notifications
- Hub sends FCM (Android) / APNs (iOS) push tokens via local WiFi when remote relay is active
- Notifications are local-first: "Motion detected in living room" is composed on hub, sent via local WiFi broadcast when app is on same network
- Remote push only when cloud relay is enabled

## Offline Sync Strategy

### App State
- App caches full device list and last-known state in local SQLite
- App can display "last known state" for all devices when offline from hub
- Offline indicator shown in app header when hub unreachable

### Offline Commands
- User can issue commands while offline — they queue in local SQLite
- When hub reconnects, queued commands are replayed in order with timestamps
- Conflict resolution: hub state wins (device may have been controlled by another user or physical switch)

### Sync Protocol
```
1. App connects to hub (LAN or relay)
2. App sends last_sync_timestamp
3. Hub returns all state changes since last_sync_timestamp
4. App applies delta to local cache
5. Hub sends current timestamp → new last_sync_timestamp
```

### Data Not Synced (Local Only)
- Voice command history
- Audit logs
- Learned patterns (stored on hub, not replicated to app)
- Household member PIN hashes

## Architecture

```
┌────────────────────────────────────────────────────┐
│  Nestra Mobile App (Flutter)                      │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐  │
│  │ Screens    │  │ State Mgmt  │  │ Local DB   │  │
│  │ - Dashboard│  │ (Provider/  │  │ (SQLite)   │  │
│  │ - Rooms    │  │  Riverpod)  │  │            │  │
│  │ - Settings │  │            │  │            │  │
│  │ - Patterns │  └─────┬───────┘  └────────────┘  │
│  └────────────┘        │                           │
│                        │                           │
│  ┌─────────────────────┴──────────────────────┐   │
│  │           Hub Communication Layer           │   │
│  │  - mDNS discovery                          │   │
│  │  - Local API client (HTTPS)                 │   │
│  │  - Cloud relay client (WSS)                 │   │
│  │  - Offline command queue                    │   │
│  └─────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────┘
```

### State Management
- Provider (simple) or Riverpod (scalable) — both are Flutter-idiomatic
- Immutable state objects — all mutations go through reducers
- Repository pattern for hub API — app never calls HTTP directly from UI widgets

## Security

### Authentication
- 6-digit PIN per household member (stored as Argon2 hash on hub)
- PIN verified locally via `/api/v1/auth/verify` — no biometrics in v0.1
- Guest role: read-only access to device state, no control

### Transport Security
- Local: self-signed TLS cert, cert fingerprint displayed during hub pairing
- Remote: Nestra Cloud relay uses WSS with mutual TLS (device cert + user cert)
- Certificate pinning in app for cloud relay domain

### App Hardening
- No local PIN storage in app (PIN entered once, never stored)
- App data encrypted at rest (Flutter `flutter_secure_storage` uses iOS Keychain / Android Keystore)
- Screen capture disabled in app (privacy for household state)

## Consequences

### Positive
- Flutter's brand theming engine directly supports white-label requirement
- Single codebase for iOS + Android reduces QA burden
- Local-first API + offline queue ensures app works without internet
- Hub discovery via mDNS eliminates manual hub setup

### Negative
- No first-class Matter SDK in Flutter — app talks to hub, hub talks to Matter
- Dart ecosystem smaller than JS/TS for Matter protocol tooling
- Flutter app binary larger than native (~15MB vs ~5MB)

### Risks
- **Mitigation:** Matter protocol complexity lives on hub (Rust), not app — app is a thin REST/WebSocket client
- **Risk:** Flutter desktop/Web not planned for v0.1 — web app for remote dashboard is Phase 2
